## 2.8節: `DEFINE_SUM_NEW_GENERIC` / `SUM_NEW` の検証ログ

**末尾コンマの制約(実機確認):** `_Generic`の連想リストも既存マクロと同じ「先頭コンマ」パターンを踏襲できるが、リストの一番先頭に来る要素がないため、専用のプレースホルダ型(`NAME_GenericPlaceholder_`)を1個手動で置いて帳尻を合わせている。末尾コンマが許されないことは実機で確認済み(`_Generic((x), TagA: f_a, TagB: f_b,)` は `expected specifier-qualifier-list before ')' token` でエラーになった)。

**型の完全一致制約とその実害(実機確認):** `_Generic`は制御式の型が完全一致しないと選択できず、暗黙の型変換に頼れない。

- `double`型のvariantに`int`型の変数を渡すと、リテラル(`3`, `3.0`)は正しく振り分けられるが、変数は暗黙昇格が効かず`selector of type 'int' is not compatible with any association`でエラーになる。
- ポインタが指す先のconst有無も別の型として扱われる。`const char*`を連想キーにすると文字列リテラル(減衰後`char*`)がマッチしない一方、`char*`をキーにすると最初から`const char*`と宣言された変数がマッチしない。両対応するには`char*:`と`const char*:`の両方を連想リストに書く必要がある。

**配列型に関する記載の誤り(訂正済み):** 当初「配列型はそもそも連想キーとして書けない(`int[3]:`は文法違反でエラー)」と記載していたが、これは未検証のまま書いた誤った記述だった。実機(`gcc 11.4.0`, `-std=c11 -pedantic-errors -Wall -Wextra`)で`_Generic(a, int[3]: 1, default: 0)`を試したところ、**エラーにならずコンパイルが通った**。ISO C11 6.5.1.1p3は「連想の型名は配列型を指定してはならない(shall not)」という制約であり、5.1.1.3の規定によれば制約違反には診断が義務付けられているが、gccはこの制約を診断していない。ただし実害はない。制御式(配列を渡した実引数)は型決定の際に配列→ポインタの読み替えを受けるため`int*`相当として扱われる一方、連想キー側の`int[3]`はそのまま配列型として扱われ両者は一致せず、`int[3]:`のブランチは常にdefaultに落ちる「絶対に選択されない死んだブランチ」になるだけで終わる。「検証していない主張をそのまま書いていた」という点は率直に記録しておく。

**型の使い回しの検出タイミング(当初の誤解を訂正):** 2つ以上のvariantが同一のペイロード型を共有していると、`_Generic`の連想リストが重複しコンパイルエラーになる。当初「`DEFINE_SUM_NEW_GENERIC`を呼んだ時点で検出される」と考えていたが、これは誤りだった。`DEFINE_SUM_NEW_GENERIC`は補助の型・関数を生成するだけで`_Generic`という構文自体をまだ展開しないため、実際に`SUM_NEW(NAME, VARIANTS, x)`を呼んで`_Generic`が展開された箇所で初めて検出される。`test_compile_guarantees.sh`のケース8bを書く過程で、`DEFINE_SUM_NEW_GENERIC`単体しか呼ばないテストコードが期待に反してコンパイル成功してしまうことに気づき、原因を調査してこの理解に訂正した。

**C11限定機能をCIに乗せる実務コスト(実機確認):** `#if defined(__STDC_VERSION__) && __STDC_VERSION__ >= 201112L`でヘッダ側をガードするだけでは不十分で、`SUM_NEW`を実際に呼ぶ利用側のファイル(examples/tests)がC99ビルドで壊れることを実機で確認した(`implicit declaration of function 'MySum_new'`)。対策として、`generic_ctor_demo.c` / `test_sum_new_generic.c` はファイル全体を同じ`__STDC_VERSION__`ガードで分岐させ、C99実行時は代替メッセージを出すだけにする構成にした。

**`NAME_new(x)`ショートカットマクロの検証(訂正済み):** 当初「ライブラリ全体の最低要求をC11に引き上げれば解決するかもしれない」としていたが、これは誤りだった。`DEFINE_SUM_NEW_GENERIC`が`NAME_new(x)`という専用マクロを生成できない根本原因は「マクロ展開結果から新しい`#define`を起こせない」というプリプロセッサの制約であり、これはC11/C17でも変わらず成立する。C標準のバージョンとは無関係の話だった。一方、この制約は「ライブラリが自動生成できない」という話であって、「利用側が自分で書くことすら制限されているわけではない」ことを見落としていた。実機で確認した通り、`DEFINE_SUM_NEW_GENERIC(NAME, VARIANTS)`の直後に利用側が

```c
#define NAME_new(x) SUM_NEW(NAME, VARIANTS, x)
```

と1行書けば(`NAME`/`VARIANTS`を実際の識別子に置き換えて)、`NAME_new(x)`という短い記法がC99/C11双方で問題なく動作する(この`#define`はユーザーが自分のソースファイルに直接書く通常のディレクティブであり、マクロ展開結果から生成されたものではない)。この一行イディオムはapi_reference.md 9節の使用例に反映済み。

## 3節: ハンドラの渡し順取り違えの検証ログ

当初は「ハンドラの中身の型さえ合っていれば、渡す順序を間違えても検出できない」と考えていたが、実際に検証したところ**これは半分だけ正しい**。各ハンドラの型は `RetType (*)(Type1*)` のように **variantごとのペイロード型で固有** になっているため、2つのvariantのペイロード型が異なっていれば、順序を入れ替えて渡すと `-Wincompatible-pointer-types` が発生し、`-Werror` 運用下ではコンパイルエラーになる(`CmdAttack` と `CmdHeal` のように**構造的に同一レイアウトでも `typedef` が別名である限り別の型として扱われる**ため、C言語の名前的型付けの恩恵で検出できることを実機で確認した)。

検出できない残存ケースは、**2つ以上のvariantが文字通り同一の型(例えば両方とも素の `int32_t`)をペイロードに使っている場合**に限られる。この場合は関数ポインタの型も完全に一致してしまうため、順序を入れ替えてもコンパイラは区別できない(実機で確認済み)。

## 4.5節: TYPEに複雑な型を渡した場合の検証ログ

**当初の想定は不正確だった。** 「コンマがマクロ引数の区切りとして解釈され展開が壊れる」という説明を疑い、`gcc -E` でプリプロセッサ展開結果だけを確認したところ、`void (*)(int,int)` のようなコンマを含む型でも `void (*)(int,int) cb;` という形でTYPEはひとかたまりのまま正しく渡っていた。プリプロセッサはマクロ引数を走査する際に括弧のネストを追跡するため、`(int,int)` のように括弧で保護されたコンマは引数の区切りとして扱われない。**これはgccの実装依存の挙動ではなく、ISO C 6.10.3 Macro replacement(C11 p.166-173 / C99 p.151-158)が規定する関数マクロの引数走査規則そのものである。**関数マクロ呼び出しの引数列は、対応する閉じ括弧`)`が現れるまでを1回の呼び出しとして切り出すが、その走査中に現れる「対応の取れた」左右の括弧のペアはスキップして中身を素通しする、という規則が明文化されている。したがって `(int,int)` のように内側で閉じている括弧の中のコンマは、外側のマクロ引数区切りとして解釈されないことが規格上保証された動作であり、gcc固有の親切な実装ではない。

実際に壊れる原因は別にある。`SUM_UNION_MEMBER`/`SUM_CTOR`/`SUM_GETTER` が生成する宣言は `TYPE 識別子;` という「型名を前に置き識別子をそのまま続ける」パターンを前提にしている。これは構造体・基本型・単純なポインタ型には有効だが、関数ポインタ型は宣言子の構文上、識別子を型の内側(`TYPE (*識別子)(引数型...)`)に埋め込む必要があるため、このパターンに当てはまらない。実機で確認したエラーは次の通り。

```
raw_fnptr.c:3:75: error: expected identifier or ‘(’ before ‘)’ token
    3 | #define V(X, NAME, EXTRA) X(NAME, EXTRA, a, PA) X(NAME, EXTRA, cb, void (*)(int,int))
      |                                                                           ^
```

つまりこれは「マクロのコンマ分割問題」ではなく、**「TYPE 識別子の単純連結パターンが、関数ポインタ型のC宣言子構文と噛み合わない」という、より一般的な制約**である。対策(typedefで単一トークンの型名にしてから渡す)は結果的に同じだが、原因の理解を訂正しておく。

`typedef`後の型名(`Callback`)は普通の識別子なので `Callback cb;` は正しく展開される。`gcc -std=c99/c11/c17 -pedantic -Wall -Wextra -Werror` すべての組み合わせでコンパイル・実行成功を確認済みで、`tests/test_compile_guarantees.sh` にケース5(typedefなしの関数ポインタ型→コンパイル失敗)・ケース6(typedef後→コンパイル成功)として回帰テスト化した。

**配列型での実機検証(当初「未検証」としていた点を確認):** 関数ポインタ型と同様、`int[3] member;`という宣言はそもそも文法違反になる。

```
t1.c:10:19: error: expected identifier or '[' token
   10 | #define TYPE_A int[3]
      |                   ^
t1.c:13:5: note: in expansion of macro 'TYPE_A'
```

`typedef int IntArr3[3];`として単一トークンの型名にする回避策は、構造体メンバー宣言(`SUM_UNION_MEMBER`が生成する`TYPE TAG;`)としては関数ポインタ型と同じく機能することを確認した(`gcc -std=c11 -pedantic -Wall -Wextra -Werror`でコンパイル・実行成功)。

しかし関数ポインタ型と違い、**配列型はtypedef経由でも`SUM_NEW_CTOR`が生成するコンストラクタ(`TYPE v`という関数引数)としては正しく機能しない**、というより踏み込んだ問題があることが分かった。C言語の関数引数では配列型は暗黙にポインタ型へ読み替えられる(C11 6.7.6.3p7)ため、`typedef`で`IntArr3`という名前を与えても、コンストラクタ内部の`v`は実際には「呼び出し元の配列の先頭要素へのポインタ」でしかなく、`sizeof(TYPE)`(=12バイト)を前提にした`memcpy(&r.as.TAG, &v, sizeof(TYPE))`のようなコピー処理は`&v`(ポインタ変数自身のアドレス、8バイト)からの範囲外読み出しになり、実機で確認したところ実際に不定値が返った。関数ポインタ型は「1個のポインタ値」として値渡し・値コピーがそのまま成立するのに対し、配列型は**そもそもC言語において代入・値渡し・値の戻り値のいずれもできない型**(6.3.2.1で規定される「配列は左辺値になれない/ポインタに読み替えられる」という一般規則)であるため、typedefで宣言構文の問題を回避しても、ライブラリが前提とする「`TYPE`を値として代入・複製できる」という設計全体の前提に配列型は合致しない。

対して`struct { int v[3]; } IntArr3Box;`のように配列をstructで包んだ場合は、struct自体が通常の値セマンティクス(代入・値渡し・値の戻り値)を持つため、`r.as.arr = v;`という構造体代入だけで配列メンバも含めて正しく値コピーされることを実機で確認した(コピー後に元の変数を書き換えても複製先に影響しないことまで確認済み)。

## 4.7節: `DEFINE_SUM_MATCH` に `RetType = void` を渡す件の訂正ログ

実際に検証したところ、`DEFINE_SUM_MATCH(..., void)` は多くの環境でコンパイルは通るが、`-std=c11 -pedantic` を付けると `ISO C forbids 'return' with expression, in function returning void` という警告が出る(`case NAME_tag: return handler();` の `handler()` がvoidを返す関数呼び出しのため)。

**訂正: 「グレーな書き方」ではなく明確な規格違反(ただしコンパイラは大目に見ている)。** ISO C 6.8.6.4 The return statement(C11 p.154 / C99 p.139 / C17 p.111-112)は、`return expression;` の形は関数の戻り値型がvoidでない場合にのみ、`return;` の形は戻り値型がvoidの場合にのみ有効、と明確に制約(constraint)として規定している。つまり「voidを返す関数の中で式付きreturnを書く」こと自体が制約違反であり、規格上は「グレー」ではなく「診断メッセージが要求される違反」である。実際にgccが`-pedantic`で警告を出すのはこの制約違反を検出しているためであり、`-pedantic-errors`を付ければハードエラーになる。今回`-pedantic`単体では警告止まりでビルドが通ってしまったのは、GCC/Clangが「診断は出すが既定ではエラーにしない」という緩い運用をしているからであり、規格の要求自体が緩いわけではない。design_spec筆者(私)の当初の理解が実際の規格の強さを過小評価していた点を訂正する。

## 4.10節: 複数翻訳単位での共有の検証ログ

**検証内容:** `command_type.h`(インクルードガード付き)で `DEFINE_SUM_TYPE`/`DEFINE_SUM_DISPATCH` を1回定義し、`worker.c`(`Command`を値渡しで受け取り`Command_dispatch`する関数を定義)と`main.c`(`Command`値を構築し`worker.c`の関数を呼ぶ)の2つの翻訳単位を個別に`-c`コンパイルしてからリンク・実行した。`gcc -std=c99/c11/c17 -pedantic -Wall -Wextra -Werror` いずれの組み合わせでもエラーなくビルドでき、実行結果も期待通りだった。念のため`-flto`(リンク時最適化。TU間の不整合をより厳格に検出しうる)でも問題なくビルド・実行できることを確認した。`tests/test_compile_guarantees.sh` にケース7として回帰テスト化した。

**理論的な注意点(規格条文の確認により記述を訂正):** 当初「匿名構造体を複数TUにまたがって使う場合の扱いは規格上グレーである」と書いていたが、ISO C 6.2.7 Compatible type and composite type(C11 p.47-48 / C99 p.40-41)を確認したところ、より正確な言い方は「グレー」ではなく「そもそも適用条件が発生しない」である。

6.2.7の翻訳単位間の型互換性規定は、構造体・共用体・列挙型については**タグによる対応付け**を前提として組み立てられている(「一方がタグ付きで宣言されていれば、他方も同じタグで宣言されていなければならない」という要件が明記されている)。この規定が実際に働く場面は、`extern struct S *x;` のように**外部リンケージを持つ識別子(オブジェクトまたは関数)が複数のTUにまたがって宣言され、その型が食い違っていないかをリンカ相当の仕組みが突き合わせる**必要があるケースである。逆に言えば、タグなし(匿名)の構造体・共用体は対応付けの手がかりとなるタグを持たないため、6.2.7の互換性規定が「同じ型として扱ってよい」と積極的に保証する対象にも、「別の型として扱う」と否定する対象にも、明確には該当しない — これが「グレー」に見えていた理由である。

しかし本ライブラリの設計では、この疑問自体が発生しない。6.2.7の互換性判定は「外部リンケージを持つ識別子の宣言がTUをまたいで一致しているか」を確認するために必要になるものであり、`DEFINE_SUM_TYPE`/`DEFINE_SUM_DISPATCH`等が生成する関数はすべて `static inline`(内部リンケージ)であって、TU間で共有される外部シンボルが一つも存在しない。型自体もリンカが照合する対象ではない。したがって「複数TUで生成された匿名union/structの型が6.2.7上互換とみなされるか」という問い自体が、本ライブラリの使い方の範囲では**問う必要のない問い**になっている。「実務上はおそらく大丈夫」という経験則的な結論ではなく、「6.2.7が要求する前提条件(外部リンケージを持つ識別子の共有)がそもそも成立しない」という設計上の理由により、この論点を安全に迂回できている、というのがより正確な説明である。共通ヘッダ経由で全TUが字面上完全に同一のコードを生成する点、および今回のLTO検証で警告・エラーが出なかった点は、この設計上の帰結を裏付ける実測結果として引き続き有効である。

## 4.12節: C言語規格書との突き合わせ検証(方法論)

4.1〜4.11節はこれまで実機での動作確認(gcc/clangでの実際のコンパイル・実行結果)を根拠にしてきたが、実機確認は「手元の環境でたまたま動いている」ことしか示せず、「規格が保証しているのか、単にコンパイラの実装がそうなっているだけなのか」を区別できない。この区別自体が重要な設計判断根拠になる(例: 4.5節の型連結パターンの限界は規格の宣言子構文そのものに起因するので将来もどのコンパイラでも変わらないが、4.6節のコンパイラ拡張はいつでも変わりうる)ため、この検証を行った。

**情報源についての正直な報告:** 当初はISO/IEC 9899:2011(N1570)のPDFを直接取得して該当条項を読む方針を試みたが、`web_fetch`によるPDF全文取得は約100KB(規格書の最初の30ページ程度、5.2.4.2.2 環境限界値の途中まで)で打ち切られ、必要な条項(6.2.7, 6.5.2.5, 6.7.2.1, 6.7.4, 6.7.8/6.7.9, 6.8.6.4, 6.10.3はいずれもp.47以降)には到達できなかった。そのため、規格の条項番号・ページ番号・条文の要旨を明示的に引用しているcppreference.com(C99/C11/C17/C89の各版のページ番号を版ごとに個別記載しているページ)を二次資料として用い、そこに記載された条項番号を手がかりに検証した。**規格原文を一字一句確認できたわけではない**ことを明記しておく。より高い確実性が必要な箇所(契約・法務用途等)では、該当版のPDF/購入版原文を直接参照すること。

| 4.10 | 匿名構造体の複数TU間互換性は「規格上グレー」 | 6.2.7 Compatible type and composite type | **主張を訂正・精緻化**。「グレー」ではなく「タグによる対応付けが前提の規定であり、匿名型には適用条件自体が発生しない」。かつ本ライブラリは外部リンケージシンボルを持たないため、この論点自体を実務上回避できている |
| 冒頭 | 対応C標準はC99以降(`static inline`、指示付き初期化) | 6.7.4 Function specifiers / 6.7.8(C99)・6.7.9(C11以降) Initialization | **主張は正確**。両機能とも規格上(since C99)と明記されている |
| 2.8 | 配列型は`_Generic`の連想キーに書くと文法違反でエラーになる | 6.5.1.1p3 Generic selection / 5.1.1.3 Diagnostics | **主張を訂正**。規格上は配列型を連想の型名にすることが制約違反であり診断が義務(shall)だが、gcc 11.4.0は実際には診断しない。「規格が保証する動作」と「手元のコンパイラの実際の挙動」が一致しない具体例として記録 |

**この検証作業から得られた教訓:** 実機での動作確認と規格条文の確認は、互いに代替できない別の検証軸である。実機確認は「今のコンパイラでは動く」ことを、規格確認は「なぜ動く/動かないと言えるのか、将来も変わらない性質かどうか」を担保する。4.7節・4.10節のように、実機確認だけでは「グレー」「非推奨」と曖昧にぼかしていた箇所が、規格条文まで遡ることで「明確な制約違反だが実害はない」「そもそも規定の対象外」という、より解像度の高い(かつ実務上の結論は変えない)説明に置き換えられた。

## 6節: CI導入の経緯

実際にCIを回したところ、ローカル(gcc・c11のみ)では検出できなかったclang固有の警告(`-Wunused-function`。7節参照)が初回実行で捕捉された。この経験から「ドキュメントに対応範囲を書くだけでは検証にならない、実際にその組み合わせでビルドするまで本当には確認できていない」という教訓を得て、コンパイラだけでなくC標準バージョンについても同様にCIマトリクスへ組み込んだ。結果として、C99が実際にビルドを通ることを継続的に保証できるようになり、design_spec.md冒頭の「対応C標準: C99以降」という記述はドキュメント上の主張ではなくCIによって裏付けられた事実になった。

## 7節: 見つかったバグと修正の記録

- **`NULL`未定義バグ:** `generic_sum_type.h` が `NULL` を使用しているにもかかわらず `<stddef.h>` を include していなかった。単体では `<stdint.h>` 等の副作用で `NULL` が見えてしまうことがあり見過ごされやすいが、`gcc -std=c11 -pedantic -Wall -Wextra -Werror` の単体コンパイルで顕在化した。`#include <stddef.h>` を追加して修正済み。
- **ロック解放漏れバグ:** `SUM_CTX_LOCK`/`SUM_CTX_UNLOCK` フックを実装した際、当初 `SUM_DISPATCH_CASE` が各caseの末尾で `return;` していたため、`SUM_CTX_LOCK(ctx)` の後に呼ばれるはずの `SUM_CTX_UNLOCK(ctx)` がswitch内の早期returnによってスキップされてしまうバグを実装直後の目視レビューで発見した。`return;` を `break;` に変更し、switch文を正常に抜けてから `SUM_CTX_UNLOCK` が呼ばれるように修正した。**もしこのままリリースしていれば、ロックを取得したまま解放しない致命的なデッドロックバグになっていた。** ロック機構を追加する際は「早期returnがcleanupコードをスキップしないか」を必ず確認すべき、という一般的な教訓として記録しておく。
- **clangでの`-Wunused-function`バグ:** GitHub ActionsのCIを初めて回した際、clangジョブのみが `DEFINE_SUM_TYPE` が生成するgetter関数(例: `Command_get_move`)に対して `-Wunused-function` エラーで失敗した。GCCは未使用の `static inline` 関数を警告しないが、Clangは警告するという実在するコンパイラ間差異が原因。生成される全 `static inline` 関数に `__attribute__((unused))`(GCC/Clang共通の拡張)を付与して修正した。**ローカル検証がgccのみだったため、この差異はCIを実際に回すまで発見できなかった**。これはこのライブラリ自身がGCC/Clang両対応を謳っていることの検証として、CI導入の意義を裏付ける実例になった。
- **`DEFINE_SUM_NEW_GENERIC`の検出タイミングの誤解:** 実装直後は「`DEFINE_SUM_NEW_GENERIC`を呼んだ時点で型の使い回しが検出される」と考えていたが、`test_compile_guarantees.sh`のケース8bを書く際に、`DEFINE_SUM_NEW_GENERIC`単体しか呼ばないテストコードが期待に反してコンパイル成功してしまうことに気づいた。原因は`DEFINE_SUM_NEW_GENERIC`が`_Generic`という構文自体をまだ展開しないためで、実際に`SUM_NEW`を呼んで初めて検出されることが分かった(2.8節参照)。ドキュメント・テストとも訂正済み。
- **「配列型は`_Generic`の連想キーに書くと文法違反でエラーになる」という記載の誤り:** 2.8節に当初「配列型はそもそも連想キーとして書けない(`int[3]:`は文法違反でエラー)」と記載していたが、これは未検証のまま書いた誤った記述だった。実機(`gcc 11.4.0`, `-std=c11 -pedantic-errors -Wall -Wextra`)で実際に試したところエラーにならずコンパイルが通った。ISO C11 6.5.1.1p3は配列型を連想の型名にすることを禁止する制約であり、5.1.1.3により診断が義務付けられているが、gccはこの制約を診断しない。実害はない(制御式は配列→ポインタの読み替えを受けるため、配列型そのものを書いた連想キーとは型が一致せず、単に「絶対に選択されない死んだブランチ」になるだけ)が、「検証していない主張をそのまま書いていた」という点は率直に記録しておく。あわせて4.5節の「配列型自体は未検証」としていた箇所も実機検証し、関数ポインタ型と違って**typedefだけでは配列型を安全にペイロードにできない**(関数引数境界でポインタに読み替えられ、`sizeof`を前提にしたコピー処理が壊れる)ことを確認、記述を訂正・具体化した。

## 8節: 今後の検討課題の検証ログ

### `DEFINE_SUM_DESTROY`/`DEFINE_SUM_COPY` の呼び出し忘れ自体を検出する手段(4.1節)

`__attribute__((cleanup(...)))`(GCC/Clang共通の拡張)を使えば、スコープを抜ける際に自動的にdestroy関数を呼ばせることができ、実機でも正しく動作した(`Msg_destroy`をラップした`Msg_auto_destroy`をcleanup属性で登録し、AddressSanitizer付きでリーク0件を確認)。ただしCにはmove semanticsがないため、値をコピーして別のcleanup付き変数に代入すると**両方の変数のスコープ終了時にそれぞれdestroyが呼ばれ二重解放になる**ことを実機で再現した(ASanが`double-free`として検出)。つまりこの属性は「呼び忘れ」を自動化で解消する代わりに「意図しないコピーによる二重解放」という新しいリスクを持ち込むだけであり、根本解決にはならない。

GCCの`-fanalyzer`も試したが、このsandbox環境の`gcc 11.4.0`では`char *p = malloc(16); (void)p;`のような最も単純な直接リークすら検出せず、有効性を確認できなかった(バージョン依存の可能性があり将来のgccでは変わるかもしれないが、少なくとも今回は根拠にならなかった)。clang-tidy独自チェックによる補完は引き続き選択肢として残るが、値が関数境界・構造体・コンテナをまたいで移動するケースまで正しく追跡するには結局は線形型(linear type)チェッカーの再発明が必要になり、根本的な解決が難しいという結論は変わらない。

**CBMCによる形式検証(実機Linux環境で追加実施):** このsandbox環境にはCBMC/Frama-Cが導入できなかったため(root権限なし、ネットワークもallowlist制で該当パッケージの取得先が全てブロックされていた)、`verify/cbmc_destroy_harness.c` / `verify/cbmc_verify_destroy.sh` を用意し、ユーザーの実機Linux環境(CBMC 5.95.1)で検証してもらった。ハーネスは3つの関数からなる: `safe_case`(正しく1回だけdestroy相当を呼ぶ)、`leaked_case`(destroy相当を呼び忘れる)、`double_free_case`(値コピー後にcleanup相当を2回呼ぶ、上記の二重解放シナリオを明示的な関数呼び出しとして書き下したもの)。結果は次の通り、すべて予測と一致した。

- `safe_case` → `VERIFICATION SUCCESSFUL`(0 of 115 failed)
- `leaked_case` → `VERIFICATION FAILED`、`[__CPROVER__start.memory-leak.1] ... FAILURE` でリークを検出
- `double_free_case` → `VERIFICATION FAILED`、`[destroy_text.precondition_instance.4] line 55 double free: FAILURE` で二重解放を検出

3関数ともループ・再帰を含まないため、CBMCのシンボリック実行は近似(bounded)ではなく状態空間を網羅的に探索している。つまりこの結果は「特定の入力で試したら再現した」という経験的検証(ASan)より一段強く、「このハーネスの範囲において当該性質が成り立つ/成り立たないことの証明」に相当する。ASanでの実機確認(二重解放の再現)と、独立した手法(CBMCによる網羅的探索)による結果が一致したことで、cleanup属性の危険性についての結論の確からしさが上がった。ハーネス自体は`verify/`にCIとは独立した任意検証として置いてあり、通常のビルド・CIには影響しない。

### `SUM_CTX_LOCK`/`SUM_CTX_UNLOCK` の再定義漏れ(4.8節)

`SUM_CTX_LOCK`/`SUM_CTX_UNLOCK`は「ヘッダが初めてincludeされた時点」でTUごとに確定し(`#ifndef`によるガード)、`DEFINE_SUM_DISPATCH`が生成する関数はすべて`static inline`(内部リンケージ、4.10節参照)であるため、**同じCtx型・同じSumType定義であっても、TUを分ければTUごとに異なるロック方針を持たせられる**ことを3TU構成(共通ヘッダ + ロックを本物にするTU + デフォルトno-opのままのTU)で実機確認した。したがって「複数スレッドから使うctxかどうかをコード上で目立たせる」という命名規則よりも、**「ロックが必要なDISPATCH利用と不要なDISPATCH利用を、そもそも同じ翻訳単位に混在させない」という物理的なファイル分割を運用ルールとする方が実効性が高い**(命名規則は人間のレビューにしか効かないが、TU分割は「どのファイルに書くか」という選択を強制する)。ただし逆に、この「TUごとに独立してロック方針を決められる」という性質自体が、「一度どこかでロックを有効にすれば、プログラム全体で有効になる」という誤解に基づく事故を招く可能性がある点は新たなリスクとして認識しておく必要がある。命名規則とTU分割は排他的ではなく併用できる。

### `SUM_NEW`の呼び出し記法の簡潔化(2.8節、4.13節)

当初の記述に誤りがあった。「ライブラリ全体の最低要求をC11に引き上げれば解決するかもしれない」としていたが、これは誤りだった。`DEFINE_SUM_NEW_GENERIC`が`NAME_new(x)`という専用マクロを生成できない根本原因は「マクロ展開結果から新しい`#define`を起こせない」というプリプロセッサの制約であり、これはC11/C17でも変わらず成立する。C標準のバージョンとは無関係の話だった。詳細な検証経緯は本ファイル2.8節を参照。
